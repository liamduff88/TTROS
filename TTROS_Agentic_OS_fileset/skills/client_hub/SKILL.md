# /client-hub
> Revisit: if the hub template or hosting choice (Notion/portal/GHL) changes. · Last touched: 2026-07-07

## Purpose
Every support and build client gets one hub. It makes accumulated work visible — the renewal mechanism, not an archive.

## Standard hub sections
Engagement overview · schedule links · support-channel rules and response times · call recordings/summaries (only if client approved recording) · top-3 takeaways per session · action items · decisions made · build manifest · workflows improved · open risks/blockers · next recommended build.

## Steps
1. **Create (onboarding)** — instantiate the section list above inside the client's isolation folder; link from the client entity page. Confirm folder exists first.
2. **Update (after each session/milestone)** — append takeaways, actions, decisions; tick completed items; update build manifest and workflows-improved list.
3. **Risk log** — open risks/blockers stay visible until closed with a note, never silently deleted.
4. **Renewal view** — keep "workflows improved" and "next recommended build" current; these are what Liam shows at renewal time.
5. **Receipt** — every hub update run writes a receipt.

## Never
- Store another client's material in the hub (isolation check).
- Post recordings/summaries without recorded client approval.
- Delete history — supersede, don't erase.
- Expose internal TTROS paths, prompts, or other-client references.

## Done when
Hub exists with all standard sections, linked from the entity page, current as of the last session/milestone; receipt written.
